firmware/host_build — C 側 frame packer のホストビルド

firmware/shared/source/ichiping_frame.c を MCUXpresso SDK なしで（普通の gcc で） 共有ライブラリ化するための足場。MCU と PC でフレーム形式がドリフトしていないことを バイト単位で確認する回帰検証用。
gcc / MinGW .so / .dylib / .dll 出力 ctypes 経由テスト

目的

pc/ichp_frame.py の pack_frame と ../source/ichiping_frame.c の ichp_pack_frame が、 同じ入力に対してバイト単位で同一の出力を返すことを Python の ctypes 越しに確認するための足場。

C 側と Python 側でフレーム形式がドリフトすると CRC が通らず実機通信が全滅するため、これが最も強い回帰検証になる（ヘッダ層を 100% カバー）。

ビルド要件

プラットフォームコンパイラ
Linuxgcc（apt/dnf でデフォルト）
macOSgcc or clang（Command Line Tools）
WindowsMinGW-w64（MSYS2 推奨。pacman -S mingw-w64-x86_64-gcc）

ichiping_frame.c の依存は <stdint.h>, <stddef.h>, <string.h> のみ。MCUXpresso SDK は不要。

ビルド

cd firmware/host_build
make

生成物:

  • Linux: libichp.so
  • macOS: libichp.dylib
  • Windows: ichp.dll

確認方法

このディレクトリはファームではないので、書込・実機・シリアル全部不要。PC でユニットテストを 1 本走らせるだけ。

A. ライブラリのリンクと ctypes 突合テスト（本筋）

  1. 上記 make で .so/.dylib/.dll が生成されることを確認
  2. テストを実行: 
    cd ../../pc
conda activate ichiping
python -m unittest test_ctypes_packer -v
  3. 期待出力: 
    test_pack_frame_matches_python ... ok
test_crc16_matches_python ... ok
----------------------------------------------------------------------
Ran 2 tests in 0.018s

OK

2 件のテストはそれぞれ:

  • test_pack_frame_matches_python — 同じ入力に対して C 側 ichp_pack_frame と Python 側 pack_frame がバイト単位で同一フレームを出すことを確認
  • test_crc16_matches_python — CRC-16/CCITT-FALSE の Python 側と C 側の実装一致確認（既知ベクタ + ランダム入力）

B. gcc が無い環境

libichp.so が見つからない場合、test_ctypes_packer は 自動 skip される（fail ではない）:

test_pack_frame_matches_python ... skipped 'libichp not built'

このときも test_frame_format（ヘッダ層 9 件）と test_loopback（E2E 6 件）は引き続き通る。最低限のフレーム整合性は保証されるが、実機を繋ぐ前に host_build までは通しておくのを推奨。

C. クリーン

make clean
