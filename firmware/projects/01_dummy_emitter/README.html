01_dummy_emitter — シリアル疎通テスト

MCU 内で合成した chirp+残響をフレーム化し、OpenSDA UART (921600 bps) で PC に流す v0.1 の本番ファーム。
v0.1 本番 921600 bps シリアル lpuart/polling_transfer ベース

MCUXpresso でのインポート手順

  1. File > New > Create a new C/C++ Project
  2. SDK Wizard → ボード frdmmcxn947 → ベース driver_examples > lpuart > polling_transfer
  3. プロジェクト名を ichiping_dummy_emitter などに変更
  4. 生成された main.c を本フォルダの main.c に置換
  5. 以下をプロジェクトに追加（リンクでもコピーでも可）:
    • ../../shared/source/ichiping_frame.c
    • ../../shared/source/dummy_audio.c
  6. Project > Properties > C/C++ Build > Settings > MCU C Compiler > Includes に追加: 
    "${ProjDirPath}/../../firmware/shared/include"
  7. ビルド → OpenSDA で書込

シリアル設定

項目値
ポートOpenSDA Debug VCP（デバイスマネージャの「COMx」）
ボーレート921600 bps
フォーマットバイナリ ICHP フレーム、8N1 フロー制御なし
受け側pc/receiver.py

動作

3 秒周期で:

  1. INFER LED ON
  2. seq から決定論的なサーボ角 5 個を生成
  3. dummy_audio_generate() で 32000 サンプルの chirp+残響
  4. ichp_pack_frame() で 64038 B のフレーム化
  5. LPUART_WriteBlocking() で送出 (~556 ms)
  6. INFER LED OFF
  7. 待機

詳細シーケンスは firmware ガイド §動作シーケンス（Dummy emitter） を参照。

確認方法

⚠ 注意: このファームは バイナリフレームを吐くので、TeraTerm / PuTTY / VS Code Serial Monitor で開いても文字化けして見えるだけ。必ず pc/receiver.py で受けること。

A. PC 側受信スクリプトで実機フレームを受け取る（本筋）

  1. conda 環境を有効化（初回のみ conda env create -f environment.yml）: 
    cd pc
conda activate ichiping
  2. デバイスマネージャで FRDM-MCXN947 の OpenSDA 仮想 COM 番号（例 COM7）を確認
  3. 受信スクリプトを起動: 
    python receiver.py --port COM7 --baud 921600 --out ../captures
  4. 3 秒ごとに 1 行ログが出れば成功。期待ログ例: 
    [     0] t=    3001ms sr=16000 N=32000 servos=[ 17.0, 65.0, 82.0,  3.0, 41.0] CRC=OK (0.33 fps)
[     1] t=    6002ms sr=16000 N=32000 servos=[ 33.0, 12.0,  7.0, 88.0, 56.0] CRC=OK (0.33 fps)
  5. 成功条件:
    • CRC=OK が連続で出る
    • ../captures/labels.csv に行が追記される
    • ../captures/frame_000000.wav ... が増えていく（メディアプレーヤで開けば chirp の "ピュッ" + ノイズ尾が聞こえる）

B. verify.py で 8 項目自動チェック（推奨）

受信ストリームが本当にフレーム形式仕様に従っているかを 8 項目で機械的に判定する：

python verify.py --port COM7 --frames 20 --strict

seq 単調増加、CRC 全数 OK、サンプル数固定、chirp の整合フィルタピーク位置などをまとめてチェック。1 件でも FAIL なら exit 1。

C. PC 受信パイプラインの単体テスト（実機なしで先に通す）

ハードを繋ぐ前に、PC 側の受信・パッカー・CRC 実装が壊れていないことを単体テストで確認しておく：

cd pc
python -m unittest test_frame_format test_loopback -v
テスト件数カバレッジ
test_frame_format9ヘッダ長 36 B、各フィールドのバイトオフセット、CRC-16/CCITT-FALSE の既知ベクタ、pack→unpack ラウンドトリップ
test_loopback6emulator.py が書いた偽フレームを receiver.py で E2E 解釈、chirp の構造、サーボ角の C 互換性

gcc/MinGW があれば test_ctypes_packer も走り、C 側 ichp_pack_frame と Python 側 pack_frame のバイト一致を 2 件で検証（無ければ自動 skip）。

D. 実機なしで loopback テスト

MCU が手元に無い段階でも、emulator.py で同じフレーム形式の偽ストリームを生成して receiver.py を通せる:

# Terminal 1: 偽 MCU が TCP で配信
python emulator.py --tcp 127.0.0.1:5000 --frames 10

# Terminal 2: 受信
python receiver.py --tcp 127.0.0.1:5000 --out ../captures_dummy

または file 経由:

python emulator.py --out fake_stream.bin --frames 10
python receiver.py --in fake_stream.bin --out ../captures_dummy

E. OpenSDA 起動ログを補助的に確認

main.c の冒頭に PRINTF("IchiPing dummy emitter: ...\r\n") が 1 行だけ流れる（バイナリ送信前）。テキストモニタで一瞬だけ見えるかは速度的に怪しいので、これは "完全に死んでないか" の最後の砦として扱う:

# TeraTerm / PuTTY を 921600 bps で開いて起動直後だけ見える可能性がある
# ただし起動から数 ms 後にはバイナリフレームが流れ込んでくるので化ける

本筋の確認は A/B/C/D で行う。
